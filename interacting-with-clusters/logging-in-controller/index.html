Logging in to the Controller

Clusters use one node, called a controller, to delegate tasks to compute nodes so they can carry out commands and complete jobs. You can complete many tasks on the PW platform after logging in to the controller, such as submitting jobs and transferring data.

There are multiple ways to log in to the controller.

With a Resource Nameâ€‹

After youâ€™ve started a cluster, copy its name from the Computing Resources box.

Expand the IDE by clicking the icon. If a terminal doesnâ€™t appear in the IDE automatically, you can open one by selecting Terminal > New Terminal. By default, the terminal will show username@pw-user-username:~/pw$.

Enter the command ssh ResourceName.clusters.pw. The terminal will display the message Warning: Permanently added 'ResourceName.clusters.pw' (ECDSA) to the list of known hosts as well as your last login and location.

You can ignore messages about failed login attempts.

With an IP Addressâ€‹

After youâ€™ve started a cluster, click on its IP address in the Computing Resources box to copy it.

With a Remote Deviceâ€‹

If you want to access a cluster from the terminal or command line on your computer, you can use an SSH key to log in to the controller (see What if I don't have an SSH key? below if you need help).

While your cluster is off, navigate to your clusterâ€™s configuration settings by clicking its gear icon in the Computing Resources box or by clicking its name in the Resources tab.

Paste your public SSH key into the Access Public Key dialog box and select Save Resource.

Now you can start the cluster. Once the cluster is active, navigate to the terminal or command line on your device and enter the clusterâ€™s IP address with ssh username@IPaddress.

Youâ€™ll receive the message Are you sure you want to continue connecting (yes/no/[fingerprint])?.

Enter yes. The terminal or command line will display Warning: Permanently added 'ResourceName.clusters.pw' (ECDSA) to the list of known hosts as well as your last login and location.

Now you can send commands to the controller from your remote device.

You can ignore messages about failed login attempts.

Why don't I have to use an SSH key for a cluster's name or IP address?â€‹

Each user's container has a private key in it, so logging in to the controller through the IDE doesn't require SSH key management. That private key is automatically propagated to a cluster whenever you start one. You can access your private container key at any time with the command cat ~/.ssh/pw_id_rsa in an IDE terminal.
