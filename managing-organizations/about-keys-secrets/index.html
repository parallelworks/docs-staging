About CSP Keys and Secrets

Each of the three main CSPs use keys and secret keys that function similarly to SSH keys. This page goes over key-sharing best practices, including how to create keys and safely manage access between users.

Best Practices for Sharing Keys​

It’s often useful to share service account keys between individuals in the PW organization.  For example, an employee with cloud console access may want to share a service account key with an intern, but the intern shouldn’t be granted console access.

Below is an example for how to share keys without actually transferring the whole key itself. This process is based on Google’s best practice documentation.

Create Public/Private Key Pair & Certificate​

It’s not a good idea to share keys among users. Instead, a user that wants to access a service account but does not have cloud console access should create a public/private key pair, then send the public key to someone who does have cloud console access and can associate the key with a service account. Then the original key owner can use the private key to access the service account.

Google suggests using OpenSSL to locally create the publick/private key pair:

openssl req -x509 -nodes -newkey rsa:2048 -days 365 \

    -keyout /path/to/private_key.pem \

    -out /path/to/public_key.pem \

    -subj "/CN=unused"

For additional security, the user should also create a self-signed certificate. This certificate holds the public key along with other important information such as the target machine’s FQDN, country and city of access, email, and company details. In other cases, this should be signed by a Certificate Authority.

To create a certificate request in OpenSSL , enter:

openssl req -new -key /path/to/private_key.pem \
    -out /path/to/certificate_request.csr

Please note that the certificate request was created with a .key file.

To self-sign the certificate in OpenSSL, enter:

openssl req -text -in /path/to/certificate_request.csr -noout -verify
openssl x509 -in /path/to/certificate_request.csr -out \ 
    /path/to/certificate.crt -req -signkey /path/to/private_key.pem \ 
    -days 365

Upload the Public Key to a Service Account​

Please note that this step was completed with the GCE Console, but you can also follow a CLI approach. 

If using the GCE Console, you need to note the private_key_id and send that back to the person who generated the public/private key pair.  The other data necessary to populate the key is the project_id, client_id (a unique ID associated with the service account), and client_email (the service account e-mail), all displayed in the GCE Console.

Authenticate With the Private Key​

This key generation was done in a Linux Docker container running on Windows. Another way to get OpenSSL on Windows is to install it via Git for Windows. It’s also possible to authenticate via the Windows command line if you already have glcloud installed.

The private key needs to be first wrapped in a .json file that holds the key as well as project information (Google’s template). This additional .json wrapping is done first, followed by the actual authentication.

To cut and paste the private key text into the .json file, note that literal \n are required. Try this command (in Linux) to change all new lines to literal "\n": tr '\n' '@' < private_key.pem | sed 's/@/\\n/g'

Cut and paste the PROJECT_ID, CLIENT_ID, PRIVATE_KEY, PRIVATE_KEY_ID, CLIENT_EMAIL into the rest of the template .json. A nearly complete example for the used-cars-test service account is below:

{
"type": "service_account",

"project_id": "modular-magpie-167320",

"private_key_id": "KEY_ID",

"private_key": "-----BEGIN PRIVATE KEY-----\nPRIVATE_KEY\n-----END PRIVATE KEY-----\n",

"client_email": "used-cars-test@modular-magpie-167320.iam.gserviceaccount.com",

"client_id": "106279635271222952172",

"auth_uri": "https://accounts.google.com/o/oauth2/auth",

"token_uri": "https://oauth2.googleapis.com/token",

"auth_provider_x509_cert_url": "https://www.googleapis.com/oauth2/v1/certs",

"client_x509_cert_url": "https://www.googleapis.com/robot/v1/metadata/x509/used-cars-test%40modular-magpie-167320.iam.gse

rviceaccount.com"

}
Note

If your private key file contains ----BEGIN RSA PRIVATE KEY----- and -----END RSA PRIVATE KEY-----, you must have it in the same format in the .json file.

Please also note that Google will reject the .json file as a “bad format” if the .json keys are capitalized (e.g. Client_email instead of client_email).

Authenticate​

This last step is the same in PowerShell, macOS, or Linux. Enter:

gcloud auth activate-service-account <CLIENT_EMAIL> 
--key-file=<key_file.json>
